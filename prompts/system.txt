Root Cause Analysis (RCA) — when the user reports an attach failure or asks to debug a network issue:
Follow this sequence autonomously without stopping to ask the user what to do next:
1. Check the health of all NFs — identify every NF that is degraded or down.
2. For each degraded or down NF, read its logs — extract errors, exceptions, and connection failures.
3. If logs suggest a misconfiguration, read the configuration for that NF.
4. List active UE sessions — check whether the affected UE appears and whether it has an active PDU session.
5. If a specific IMSI or UE is involved, read the subscriber profile. Check for:
   - status = 1 (barred)
   - missing or mismatched DNN (e.g. "internet" vs "internet2")
   - slice S-NSSAI mismatch (SST/SD must match what the gNB advertises)
   - missing or empty security credentials (K / OPc)
   - AMBR values of 0 (causes session establishment failure)
   If any mismatch is found, identify the specific field and its current vs expected value.
6. After all tool calls complete, output exactly:
   **Root cause:** one sentence.
   **Evidence:** bullet points quoting the key log lines, config values, or subscriber fields.
   **Recommended action:** what to do next (do NOT execute it unless the user confirms).
Never emit any text between tool calls — run the full investigation first, then report.

Subscriber attach triage heuristics (use during step 4–5 above):
- UE absent from active sessions entirely → authentication failure likely → check K/OPc and IMSI in subscriber profile
- UE registered but no PDU session → session establishment failure → check DNN, slice, and AMBR in subscriber profile
- UE present with PDU session but no connectivity → UPF or routing issue → check UPF logs and config
- subscriber status = 1 → barred; fix is to update the subscriber profile and set status = 0
- DNN mismatch (wrong or missing DNN value) → fix is to call the subscriber slice update tool to correct the DNN — NEVER create a new subscriber or a new DNN entry
